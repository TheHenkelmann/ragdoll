# Deploy Ragdoll

Ragdoll is a **single-container, single-replica** application. Do not scale it horizontally: libSQL, the job queue, and local model cache assume one running instance per persistent `/data` volume.

All templates mount persistent storage at `/data`.

Default image: `ghcr.io/thehenkelmann/ragdoll:latest`

## JWT secret (`RAGDOLL_JWT_SECRET`)

**By default, every cloud deploy generates a random JWT secret automatically.** That secret is written into the container environment but **not saved or shown to you** — there is no download, email, or Secret Manager handoff.

Implications:

- You can deploy with one click and log in via the UI without preparing a secret first.
- If the container is recreated with a new deploy and you did not override the secret, **existing JWTs and API key tokens become invalid**.
- For production, set your own stable secret before deploying.

### Use your own secret (override)

```bash
export RAGDOLL_JWT_SECRET="$(openssl rand -hex 32)"   # or any strong value
```

| Deploy path | How the override is applied |
|---|---|
| GCP (`deploy/gcp/*/deploy.sh`) | Reads `RAGDOLL_JWT_SECRET` from the shell environment |
| AWS (`deploy/aws/*/deploy.sh`) | Passes `JwtSecretOverride` to CloudFormation |
| AWS (Launch Stack in console) | Parameter **JwtSecretOverride** (leave empty for auto-generated) |
| Azure (Deploy to Azure button) | Parameter **jwtSecretOverride** (leave empty for auto-generated) |

Local Docker Compose is unchanged: set `RAGDOLL_JWT_SECRET` in your environment or `.env` (default placeholder: `change-me-in-production`).

## Before you deploy

1. Publish the container image by pushing a `v*` tag (see `.github/workflows/release.yml`).
2. Set the GHCR package visibility to **public** so cloud services can pull it.
3. For production, export `RAGDOLL_JWT_SECRET` before running deploy scripts (see above).

## Serverless (scale-to-zero capable, max 1 instance)

| Cloud | Template | Notes |
| --- | --- | --- |
| GCP | [Cloud Run](gcp/cloud-run/) | Mirrors image to Artifact Registry, mounts Cloud Storage via GCS FUSE |
| Azure | [Container Apps](azure/container-apps/) | `minReplicas=maxReplicas=1`, Azure Files mount |
| AWS | [App Runner](aws/app-runner/) | Mirrors image to ECR, single instance |

## Simple instance (recommended for production DB use)

| Cloud | Template | Notes |
| --- | --- | --- |
| GCP | [Compute Engine](gcp/gce/) | Container-Optimized OS VM + persistent disk |
| Azure | [Container Instances](azure/aci/) | Single ACI group + Azure Files |
| AWS | [ECS Fargate](aws/ecs-fargate/) | `desiredCount=1`, EFS mount |

## SQLite / libSQL on network filesystems

Serverless templates use network-backed storage for convenience. For heavy production workloads prefer the simple-instance templates with block storage (GCE persistent disk) or EFS in provisioned throughput mode.

## Environment variables

See [CONFIGURATION.md](../docs/configuration.md). Minimum required values:

```text
RAGDOLL_DATA_DIR=/data
RAGDOLL_JWT_SECRET=<auto-generated on cloud deploy, or your override>
```

Also consider setting `RAGDOLL_SUPERADMIN_PW` so the default `admin` password is not used in production.
